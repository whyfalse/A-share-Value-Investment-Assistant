# A股数据获取工具探测与优先级编排

## 能做什么

真实探测本地安装的金融取数工具(MCP/本地脚本/联网搜索等)，核实各自的可用性、覆盖市场与数据类型，由 AI 按"数据类型→有序工具链(含降级兜底)"一次性编排成**具体配置**写入长期记忆。各复盘技能取数时直接读取配置照做，无需临场推断。

## 适用边界(Scope)

本skill遵守 `../../../references/boundaries.md` 的共享边界,以下为本skill特有边界:

1. **只编排"用什么工具取数",不改变"要取什么数据"**:各复盘技能的数据契约(需要哪些字段、什么口径)仍由各技能自定,本skill不碰。
2. **不做实际复盘取数**:本skill只产出/校验配置,不替复盘技能去拉行情、查公告。探测时为验证工具可用性可做最小试探调用(如查一个已知代码的最新价),但不展开任何复盘分析。
3. **诚实记录探测结果,不臆造工具能力**:某工具是否覆盖某市场/数据类型,以其 skill/MCP 描述文档或最小试探调用的实际结果为准;探测不到、未验证的能力如实标注 `unverified`,不凭印象填写。
4. **配置是"当前环境快照",会过期**:工具可能被装上/卸载/失效,故配置带 `next_review_due` 与轻量校验机制(见下文),发现失效即重新编排。

## 探测与编排执行步骤

### 第0步:确认是否需要跑

- 用户明确要求"刷新/重新探测/检查数据源配置" → 跑**完整探测模式**(下面第1-4步全做)。
- 复盘技能启动前接力调用 → 先做**轻量校验模式**(见后文"轻量校验"小节);校验通过直接返回,失败才升级到完整探测。
- 记忆里 `data-source-config.json` 不存在 → 直接跑完整探测模式建立首份配置。

### 第1步:盘点候选工具

列出当前会话里所有可能用于金融取数的能力,分三类来源:

- **专业金融数据工具**:查当前可用的 skill / MCP 列表,名称或描述含"金融/行情/财经/选股/市场数据/wind/quote/finance"等、以金融数据为专门用途者。记录每个工具:名称、调用方式(skill名 / MCP工具名 / 本地脚本路径)、**自身描述声明覆盖的市场与数据类型**。
- **本地取数脚本 / 数据包**:项目内或技能内自带的取数脚本、本地导出的行情文件、已安装的 Python 行情包等。
- **通用联网搜索**:可用的 web search / web fetch 类工具(如 `WebSearch`、`mcp__WebSearch__*`、`WebFetch`)——定位为**兜底与时效补充**,不作结构化数值首选。

### 第2步:探活与能力核实

对每个候选专业工具/脚本做**最小可用性试探**(不展开复盘):

- 专业金融工具:用一个已知标的做一次最小查询(如查某A股最新价),确认能正常返回、Key/额度/网络是否就绪。返回正常记 `available`;认证/额度/网络失败记 `blocked` 并注明原因;调不通记 `unavailable`。
- 本地脚本:确认运行时(python/node 等)与依赖是否就绪(如 `python -c "import akshare"`、`node --version`),记录版本。
- **重点核实"覆盖边界"**:很多专业工具有明确不覆盖的市场(例如某些工具不含日股/欧股/汇率/期货)。以工具描述+试探结果为准,把"覆盖哪些市场、不覆盖哪些"如实记下——这是后续编排里"该市场该走谁"的关键依据。

### 第3步:AI 编排优先级

按"数据类型"分桶,为每个桶编排一条**有序工具链**(取到即停,后面的是降级/兜底)。编排遵循 `boundaries.md` §8 的总原则**先专业、后通用**,并落实到具体工具名:

按数据类型至少覆盖以下桶(可按实际环境增减):

- **A股结构化数值**:行情/K线/分钟线、财务基本面、资金流向、估值、股东、指数成分等。
- **港股/美股结构化数值**(若有专业工具覆盖)。
- **基金/ETF/指数/债券数值**(若有专业工具覆盖)。
- **A股技术指标计算用日线**(技术引擎专用,通常走本地 akshare 脚本及其降级链)。
- **宏观经济指标**(GDP/CPI/PPI/PMI/社融/利率等,优先专业 EDB 接口)。
- **公告/财报文档**(优先专业公告接口)。
- **消息资讯/新闻/政策/事件**:专业资讯接口 + 联网搜索**组合交叉印证**,不只用单一搜索。
- **专业工具不覆盖的市场**(如日股/欧股/汇率/隔夜外盘等):明确标注"无专业源,只能联网搜索,且只采信定性信息(涨跌/方向),精确数字注明未经专业源核对"。

每条工具链给出:`primary`(首选具体工具+调用方式)、`fallbacks`(有序降级列表)、`notes`(口径/限制/兜底时的诚实声明要求)。

编排时严守两类数据的处理纪律(与 `data-source-priority.md` 一致):
- **结构化数值类**:能用专业工具就绝不用搜索凑数字;专业工具字段缺失按各技能降级规则如实声明,不编造。
- **消息资讯类**:默认多工具组合、交叉印证,以专业/官方来源为准,搜索补时效广度。

### 第4步:写入记忆

把编排结果写入 auto-memory 目录的 `data-source-config.json`(完整结构见 `references/config-schema.md`),并确保 `MEMORY.md` 有指向它的指针(没有则新增,有则保持最新)。

记忆文件路径:`<auto-memory目录>/data-source-config.json`(与 `ashare-macro-context.json` 同目录)。

写入后向用户简报:探测到哪些工具(可用/受阻/不可用)、各数据类型分别编排给了谁、有哪些市场只能走联网搜索兜底。

## 轻量校验模式(复盘技能启动前每次调用)

复盘技能启动取数前应快速核对配置是否仍然成立,避免拿着失效工具硬跑。校验只做轻量动作:

1. 读 `data-source-config.json`;**不存在** → 升级到完整探测模式。
2. 核对配置里标为 `primary` 的关键专业工具此刻是否仍在当前 skill/MCP 列表中、运行时依赖是否仍在;**关键 primary 工具消失或依赖缺失** → 升级到完整探测模式重新编排。
3. 看 `next_review_due` 是否已过期;**已过期** → 提示并升级到完整探测模式。
4. 以上都通过 → 配置有效,直接返回供复盘技能使用,不做多余试探调用。

> 校验要轻:常规情况下只读配置 + 比对工具列表即可,不必每次都对每个工具做试探调用,以免拖慢复盘启动。只有发现疑似失效迹象才升级到完整探测。
