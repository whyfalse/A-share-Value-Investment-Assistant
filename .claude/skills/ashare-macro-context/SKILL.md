# A股宏观环境背景维护

## 适用边界(Scope) —— 必须遵守

1. **只描述背景,不做信号判断**:本skill维护的所有内容是"当下国内外宏观环境的事实陈述",不对短期买卖时点做任何推断、不给买入/卖出/止盈止损建议。这条边界优先级最高,任何环节都不能突破。
2. **不涉及具体数据获取方式**:本skill不规定该调用什么搜索引擎、API或数据源,这部分由使用者或运行环境自行决定。本skill的职责仅是:**应该维护哪些维度 → 按什么频率更新 → 用什么结构存到本地**。
3. **与个股分析skill是互补关系,不是替代关系**:例如本地若存在`ashare-technical-analysis`等个股层面的分析skill,本skill提供的是它们可以引用的"宏观背景知识库",用于环境校准、防止把"长期已知的宏观背景"误判为"新催化剂"——而不是反过来用宏观背景去直接驱动个股买卖结论。

## 维度框架总览

完整的维度定义、子项、典型观测指标见 `references/dimensions.md`(首次维护或不确定某维度包含哪些子项时,务必先读这个文件,不要凭记忆假设)。**当前版本框架(详见 dimensions.md 顶部版本号)共 13 大类,但这不是封闭穷尽列表**——它是当前认为重要的维度集合,会随宏观环境演变而增减。遇到现有类别无法合理覆盖的新因素时,应走下文的"维度扩展机制",而不是硬塞进某个不贴切的类别,也不要因为"框架里没写"就忽略不记。

| 类别代码 | 类别 | 说明 |
|---|---|---|
| A | 政治与政策 | 重大政策方向、关键人事、法规变化 |
| B | 宏观经济基本面 | GDP、PMI、消费、投资、进出口、就业 |
| C | 货币与流动性 | 央行基调、利率、社融、M1/M2 |
| D | 财政政策 | 赤字率、专项债、地方债务、减税降费 |
| E | 汇率与跨境资本 | 人民币汇率、北向南向中长期趋势、外资权重 |
| F | 房地产周期 | 销售/新开工、房企融资、保交楼、限购限贷 |
| G | 通胀与大宗商品 | CPI/PPI、原油/铁矿/铜等大宗价格 |
| H | 人口与社会结构 | 出生率、老龄化、城镇调查失业率、储蓄率 |
| I | 国际形势 | 细分:中美关系(关税/出口管制/高层会晤/台海)+ 其他地缘政治 |
| J | 资本市场制度政策 | A股特色:IPO节奏、退市/减持新规、注册制、印花税 |
| K | 全球主要央行政策 | 美联储/欧央行/日央行、美元指数 |
| L | 重要会议/时间节点日历 | 两会、政治局会议、中央经济工作会议、FOMC等 |
| M | 长期产业热点与行业风口 | 强调"长期"——区别于短期题材炒作的持续性产业趋势 |

> 注:用户原始需求中的"政治、经济、政策、长期热点事件、行业风口、国际形势"已分别对应到 A/B/D/M/M/I,其余 C、E、F、G、H、J、K、L 是为完善宏观传导链条而补充的维度,缺一类都可能导致复盘时遗漏重要的环境背景。

## 维度扩展机制(Extension Mechanism)

框架需要能随宏观环境演变扩展,但又不能因为每个新鲜事件都升级成顶层类别而无限膨胀、失去结构感。处理新因素时按以下判断顺序:

1. **优先归入现有类别的子项**:默认假设新因素能放进 A-M 中某一类(例如"AI监管政策"大概率归入 I 或 M 的子项,不必新建顶层类别)。这是默认路径,大多数情况到这一步就结束。
2. **拿不准时先进"候选维度暂存区"，不要立即新增顶层类别**:如果新因素明显不属于现有任何一类,但又不确定是否值得长期单列跟踪,记一条 `candidates`(模板见 `assets/macro_context_template.json` 中的 `candidates` 数组),标注发现日期、初步理由、暂定归类建议,**不要**急着升级成正式类别,也不要因为犹豫就完全不记录。
3. **复核候选项,决定转正/合并/淘汰**:每次维护时检查 `candidates` 中是否有条目已经历过 1-2 个 tier-2 周期(约1-2个月)仍持续活跃、且确实无法被现有类别覆盖——只有满足这个条件才升级为正式顶层类别(新增 N、O...编号),并在 `references/dimensions.md` 顶部的变更记录里留痕(新增日期、原因)。若候选项逐渐降温或被证实只是短期事件,标记为 `rejected` 或 `merged_into_existing`,不删除记录,留作历史依据。
4. **新增正式类别后同步更新本文件的总览表**:既要更新 `dimensions.md`,也要回来给上面这张总览表加一行,保持两处framework定义一致,避免后续维护时两个文件描述的类别数量对不上。

这套机制的核心是:**默认收敛(优先塞进现有类别),不确定时缓冲(候选区观察一段时间),只有被证明有持续价值才真正扩张(正式新增类别)**——避免框架一边太死(13类封顶)、一边又太松(逢新闻就立项)。

## 更新频率分层(Tier System)

每条记忆必须打上 tier 标签,这样使用方能立刻判断信息新鲜度,避免把陈旧背景当成新催化剂:

- **Tier 1 慢变量(结构性)**——建议复核周期 90 天:人口结构、产业政策大方向、中美关系框架定位、资本市场制度大方向、长期产业热点的趋势判断
- **Tier 2 周期变量**——建议复核周期 30 天(或随月度数据发布):CPI/PPI、社融、M1/M2、汇率水平、房地产销售数据、央行公开市场操作力度
- **Tier 3 事件驱动变量**——发生时即时更新,不设固定复核周期,需标注具体日期:央行议息决议、重要会议结论、地缘政治突发事件、关税/出口管制变动、重大人事变动

## 本地存储结构(Local Memory Schema)

完整字段定义与示例见 `assets/macro_context_template.json`。核心原则:

- 一条记忆 = 一个维度的当前状态快照,而不是新闻流水账。更新时**覆盖式修订** `summary`,不要无限累加旧文字;如需保留环境变化轨迹,使用 `history` 数组追加而非散落新建条目。
- 必填字段:`category`(对应当前框架的类别代码)、`dimension`(具体子项)、`tier`(1/2/3)、`summary`(一句话客观陈述,不含买卖判断)、`last_updated`、`next_review_due`(tier 3 可为 null)。
- 文件顶层需要一个 `framework_version` 字段,记录当前使用的维度框架版本(对应 dimensions.md 顶部版本号),以及一个独立的 `candidates` 数组,专门存放尚未转正的候选维度(字段见 `macro_context_template.json`:`candidate_name`/`proposed_category`/`first_noted`/`rationale`/`decision_due`/`status`)。`entries`(已确立维度)和 `candidates`(候选维度)要分开存放,不要混在一起,否则无法区分"已确认的背景"和"还在观察的苗头"。
- 存储路径不做强制规定,建议使用环境约定的记忆目录(如 `~/.claude/memory/` 或项目内固定目录),文件名建议 `ashare-macro-context.json`。

## 维护工作流

当触发"更新/维护宏观背景"时,按以下步骤执行:

1. **读取现状**:读取本地现有记忆文件;若不存在,以 `assets/macro_context_template.json` 为骨架新建,当前框架版本下的类别都建空位,不要遗漏已确立的类别。
2. **判断优先级**:先处理 (a) `next_review_due` 已到期或临近的条目,(b) Tier 3 事件驱动类的新增/突发情况,(c) `candidates` 中 `decision_due` 已到期、需要决定转正/淘汰的候选项。Tier 1 的慢变量若未到期,无需强行更新。
3. **逐项核实并改写**:对每个需要更新的维度,只写"是什么"的客观事实陈述,不写"因此应该如何操作"。语言要短、要中立,一句话讲清当前状态即可,避免堆砌长段落。
4. **识别新因素,按扩展机制分流**:若发现某个新宏观因素不属于现有任何类别,先判断能否归入某类的子项;若确实归不进去,按"维度扩展机制"加入 `candidates`,不要因为框架里没有现成位置就丢弃这个信息。
5. **回写时间戳**:更新 `last_updated`;按 tier 自动推算新的 `next_review_due`(tier1 = +90天,tier2 = +30天,tier3 = null 或由事件性质另行判断,例如"下次FOMC会议日期")。
6. **保留轨迹(可选但推荐)**:把被覆盖的旧 `summary` 追加进该条目的 `history` 数组,附带原 `last_updated`,方便回溯环境是如何演变的。
7. **写回并汇总**:保存文件后,向用户输出一份简短的摘要——本次更新了哪些维度、有没有新增/转正/淘汰的候选项——而不是把整份文件原样贴出来。

## 供其他流程引用时的规则

当个股复盘等其他流程需要调用这份宏观背景时:

- 只读取与当前分析相关的维度 `summary`,不要不加筛选地把全部 13 类塞进去。
- 引用时必须明确告知:这些内容是"环境背景校准",不能替代、也不能直接决定短期(5个交易日以内)的买卖时点判断——这是不可违反的硬边界,任何调用场景都要保留这句澄清。
- 如发现某条 `next_review_due` 已严重过期(例如超期一倍以上),应提示用户"该背景信息可能已过时,建议先更新"再使用,而不是默默使用陈旧数据。
