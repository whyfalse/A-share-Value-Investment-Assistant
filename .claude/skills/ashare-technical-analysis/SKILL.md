# A股技术面分析(中长期)

本skill遵守 `../../../references/boundaries.md` 中的共享边界（不构成投资建议、最终决策与风险由用户自行承担等）。

## 这个skill做什么 / 不做什么

**做**:基于历史日线价格与成交量,计算趋势、动量、量能、波动率/位置、
相对大盘强弱五个维度的技术指标,综合给出适用于**至少5个交易日以上**
持仓决策的买卖建议,并附止损/目标位参考。

**不做,也不应该被诱导去做**:
- 不分析财务报表、估值水平、行业景气度等基本面信息
- 不分析新闻、研报、社交媒体情绪等消息面信息
- 不提供日内交易、T+0、超短线(持仓<5个交易日)信号——如果用户明确
  是要做超短线,应如实告知本skill的指标参数(年线、季线等)是为中长期
  设计的,对超短线场景参考价值有限,而不是直接套用结论
- 不支持港股、美股、期货期权、可转债、基金——如果用户提供的代码/名称
  不是A股,应明确说明本skill的局限并停止套用其结论
- 不做"精确预测未来涨多少/跌多少"——技术分析只能给方向性、区间性的
  参考,任何看起来过于精确的数字预测都应该被改写为区间或定性判断

## 工作流程

### 步骤0:确认输入

把用户给的代码/名称标准化为标准A股代码(6位数字 + 交易所,如
`600519.SH`、`000001.SZ`)。如果用户只给了股票名称没给代码,
先尝试用你已有的知识或网络搜索确认代码,代码不确定时跟用户确认一下,
避免分析错股票。如果识别出这不是A股(港股代码、美股代码、基金代码等),
直接说明本skill不支持,不要勉强套用。

### 步骤1:获取数据

这一步的目标只有一个:拿到一份**标准化的日线CSV**,供步骤2消费。
怎么拿到不限定工具——本skill自带 `scripts/fetch_data.py` 作为开箱即用
的默认方案,但**如果用户的环境里已经有更顺手的行情数据工具,优先用
用户已有的**,不要强行要求安装本skill依赖的库。常见的"已有工具"包括:

- 其它行情数据 Python 包(如 tushare、efinance、baostock、qstock、
  Wind/iFinD 的 Python API 等)
- 通过 MCP 暴露的金融数据服务(如本项目里的 `wind-mcp-skill` /
  `wind-find-finance-skill`,或其它行情类 MCP server)
- 券商/同花顺/通达信导出的本地 CSV、用户自己的数据库

判断顺序建议:先看用户是否明确指定了数据来源 → 再看环境里是否已装好
可用的行情包或挂载了行情类 MCP → 都没有时才回退到本skill自带的
`fetch_data.py`(默认走 akshare)。

**无论用哪个工具,产出必须满足下面的数据契约**,否则步骤2无法计算:

- 文件:一份 CSV(默认放 `./output/<code6>_daily.csv`)
- 必需列:`date`(YYYY-MM-DD)、`open`、`high`、`low`、`close`、`volume`
- 可选列:`amount`、`turnover`、`pct_chg`(缺失时步骤2/3会自行处理或降级)
- 价格口径:**前复权(qfq)**,保证均线/指标在除权除息后连续可比;
  若数据源只能给不复权数据,需在报告里说明这一局限
- 排序:按 `date` 升序、无重复日期;回溯长度尽量≥250个交易日(约2年),
  以保证年线等长周期指标有效
- 标的范围:个股给前复权日线;指数/ETF同样落成上述列(指数无复权)。
  指数代码与同名个股代码可能冲突(如 `000001` 既是上证指数也是平安银行),
  用非自带工具取指数时务必确认取到的是指数而非个股

如果用非自带工具取数,取完后你要么直接把它整理成上述CSV落盘,
要么口头确认列名/口径符合契约,再进入步骤2。

**默认方案——自带脚本 `scripts/fetch_data.py`**:

```bash
python scripts/fetch_data.py --code 600519 --out-dir ./output \
  [--config ./config/data_source.yaml] [--asset-type stock|index|etf] \
  [--start 2023-01-01] [--end 2025-01-01]
```

- 如果当前目录下没有 `config/data_source.yaml`,提示用户:可以复制
  `config/data_source.example.yaml` 为 `data_source.yaml` 并按需修改
  (比如填入自己的tushare token,或指向本地CSV)。如果用户没有特殊
  需求,不传 `--config` 也可以,脚本会默认仅尝试 akshare(免费、
  无需注册,但依赖公开接口,偶尔可能因接口变动而失败)。
- 运行后会输出一份JSON(同时保存为`<code>_meta.json`),仔细看
  `ok`、`flags`、`warnings`字段:
  - `ok=false`:数据获取失败,把`errors_by_source`里的报错原样告知用户,
    并给出排查建议(检查网络、`pip install -U akshare`、换数据源、
    或改用环境里已有的行情工具等),
    **不要在没有数据的情况下编造分析结论**。
  - `flags.insufficient_history` / `is_st` / `suspected_suspension_gaps` /
    `recent_limit_move_days`:这些是数据质量风险信号,会在后续打分时
    自动触发风险否决规则(见步骤3),但你也应该在最终报告里口头上
    呼应这些风险,而不是只让脚本默默处理。

**用非自带工具取数时**:`<code>_meta.json` 不会自动生成,上面那几个
数据质量风险信号(历史长度、ST、停牌缺口、近期涨跌停)也就没有脚本
替你检查。此时你要么手动构造一份等价的 meta(至少包含 `rows`、
`date_range`、是否ST、是否有长停牌缺口的判断),要么在拿到CSV后
自行做一遍同样的质量审视,并在步骤3给 `score.py` 传入对应的
`--meta-json`,避免因为换了数据源就漏掉风险否决环节。

### 步骤2:计算技术指标(`scripts/compute_indicators.py`)

```bash
python scripts/compute_indicators.py --csv ./output/600519_daily.csv \
  [--benchmark-csv ./output/benchmark_000300.csv] \
  --out-json ./output/600519_indicators.json
```

- `--benchmark-csv` 用步骤1输出meta中的`benchmark_csv_path`(如果有);
  没有的话不传这个参数,相对强弱维度会在打分阶段自动跳过。用非自带
  工具取数时,可同样准备一份基准指数(默认沪深300)的CSV——只需
  `date`、`close`两列——传进来即可启用相对强弱维度,没有就跳过。
- 输出包含`latest`(最新指标值)、`signals`(金叉死叉/背离等离散信号)、
  `key_levels`(支撑阻力位)、`recent_series`(近期走势精简序列)。
  各指标的计算口径和参数选择理由见 `references/indicators_guide.md`,
  通常不需要每次都读,遇到数字看不懂或想验证时再查。

### 步骤3:多维度打分(`scripts/score.py`)

```bash
python scripts/score.py --indicators-json ./output/600519_indicators.json \
  --meta-json ./output/600519_meta.json --out-json ./output/600519_score.json
```

- 输出五个维度的分数与理由、加权总分、结论文字、置信度、
  止损/目标位候选、风险否决信息。完整的打分规则、权重、分档标准、
  风险否决条件见 `references/scoring_framework.md`。
- 这一步已经把"要不要否决结论改为观望"这种风险判断做掉了,
  你不需要重新推导,但要在报告里把`risk_overrides`里的内容讲清楚
  ——用户需要知道"为什么这次置信度是低,即使分数看起来不小"。

### 步骤4:撰写最终报告

用步骤1-3的JSON输出,按照 `references/report_template.md` 里的
固定结构撰写中文Markdown报告。这一步**不是简单地把JSON字段抄进模板**——
你需要:
- 对趋势形态、量价关系等做出有判断力的文字描述(比如"均线尚未形成
  清晰的多头排列,但MA5已上穿MA20,如果后续放量突破MA60可能确认
  趋势反转"),而不是只甩"trend_score=0.5"这种数字
- 当五个维度互相矛盾时,明确指出矛盾本身,而不是挑顺眼的维度强调、
  弱化不顺眼的维度
- 任何止损/目标位、风险回报比都要说明这是参考区间,不是精确预测
- 报告结尾必须包含风险提示与免责声明(模板里已经写好了标准措辞,
  可以适当改写但不要删减核心要点:不构成投资建议、技术分析的局限性、
  最终决策与风险由用户自行承担)

撰写完成后,除了在对话中展示,还必须把完整报告落盘保存到
`../../../output/ashare-technical-analysis/` 文件夹下,文件名为
`<标准代码>_technical_[日期].md`(代码用步骤0标准化后的形式、去掉点号,
日期格式 `YYYY-MM-DD`,如 `600519SH_technical_2026-06-25.md`)。
若文件夹不存在则先创建,同名文件已存在时直接覆盖。注意这里保存的是
最终中文报告,与步骤1-3脚本输出到 `./output/` 的 CSV/JSON 中间产物分开。

## 几条贯穿全程的原则

**中长期视角优先**:本skill的指标参数(20/60/120/250日均线、日线级别
的MACD/RSI/KDJ)是为至少5个交易日的持仓决策设计的。如果某次分析的
均线、ADX、量能信号互相打架,大概率说明现在正处于趋势转换的模糊期——
此时"如实承认看不清"比"硬给一个方向"对用户更有价值。

**诚实地输出不确定性**:加权总分落在中性区间,或置信度为"低",
就应该明确建议观望,而不是因为用户似乎很想要一个"买"或"卖"的
答案就强行倒向一边。如果用户追问"那你倒是说能不能买啊",
可以解释"现在的信号还不够清晰,给出一个看似果断但缺乏支撑的答案,
对你的决策没有帮助",而不是放弃中性判断去迎合。

**风险优先于收益幻想**:报告的重点应该放在"风险回报比是否合理"、
"止损位设在哪里、空间多大"上,而不是渲染"目标涨幅有多诱人"。
如果用户的提问方式带有明显的情绪化交易倾向(比如已经重仓被套、
急于找理由加仓或者"扛单"),仍然给出客观的技术面信息和风险提示,
不要用确定性语言去强化其冲动决策。

**不编造数据**:任何脚本输出中缺失的字段(数据源失败、相对强弱
因基准数据缺失被跳过等),如实说明缺失,不要为了报告完整而编造数字
或假装计算过某个并未真正算出的指标。

## 引用文件

- `config/data_source.example.yaml`——自带脚本的数据源配置示例,支持
  akshare(默认)、tushare(需token)、efinance、本地CSV,用户可按需
  启用/修改;仅在使用自带 `fetch_data.py` 时需要
- `scripts/fetch_data.py`——**默认/兜底**的数据获取脚本,负责代码标准化、
  多数据源回退、数据质量检查;若环境里已有更顺手的行情工具(其它Python
  包、行情类MCP、券商导出等),可不用本脚本,自行产出符合步骤1数据契约
  的CSV即可
- `scripts/compute_indicators.py`——计算全部技术指标、支撑阻力位、
  金叉死叉、背离信号
- `scripts/score.py`——五维度加权打分、结论生成、止损止盈参考
- `references/indicators_guide.md`——各指标计算口径与中长期解读方式
  (按需查阅,排查"这个数字怎么来的"时用)
- `references/scoring_framework.md`——评分权重、分档规则、风险否决规则
  的完整说明(用户想自定义权重,或你需要解释"为什么是这个结论"时查阅)
- `references/report_template.md`——最终报告的固定结构模板与撰写原则
  (撰写步骤4报告时必读)

## 环境依赖

步骤2/3的计算脚本基于 `pandas`、`numpy`、`pyyaml`(均为常见预装库),
这几个是本skill计算指标的硬依赖。

数据获取(步骤1)的依赖则取决于你选用哪个工具,**不强制**:
- 用自带 `scripts/fetch_data.py` 的默认数据源 `akshare` 时,需要联网
  且需要执行一次 `pip install akshare`(本skill不自带,因为不同环境的
  网络/Python环境差异较大,初次使用时如果导入失败,直接告知用户安装
  命令即可)。若环境无外网,引导其用 `config/data_source.yaml` 的
  `local_csv` 选项,自行导出/上传历史K线数据。
- 用环境里已有的其它行情工具(其它Python包、行情类MCP、券商导出等)
  时,本skill不对其依赖做任何要求——只要最终产出满足步骤1的数据契约
  (标准列、前复权、按日期升序的CSV)即可。这种情况下不需要安装
  akshare,也不需要 `data_source.yaml`。
