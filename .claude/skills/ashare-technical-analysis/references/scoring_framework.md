# 评分框架说明

`scripts/score.py` 是本文档的可执行版本。这里把规则写成人话,
方便你理解"为什么是这个结论",以及在用户要求时知道可以调整哪些地方
(例如用户更看重短期动量、或想换一套权重)。

## 设计理念

单一指标(比如只看RSI,或只看一根均线)在中长期分析里很容易给出
误导性结论——比如强趋势行情中RSI可以长期超买而价格继续上涨。
所以本skill坚持"多维度交叉验证"的思路:把技术信息拆成5个相对独立的
维度,各自打分(-2到+2),再加权汇总。**任何结论都应该能说清楚
是哪几个维度支持的,而不是给一个黑箱数字。**

如果5个维度互相矛盾(比如趋势向好但动量严重超买、量能背离),
总分会落在中性区间,这时**应该如实输出"观望/信号不一致"**,
而不是为了"给用户一个痛快的答案"强行偏向买或卖。这是本skill
最重要的原则,优先级高于"让结论看起来更有用"。

## 五个维度与默认权重

| 维度 | 默认权重 | 衡量什么 |
|---|---|---|
| 趋势 trend | 35% | 均线排列、价格与年线关系、ADX趋势强度 |
| 动量 momentum | 20% | RSI/KDJ超买超卖、MACD交叉、量价背离 |
| 量能 volume | 15% | 量比、放缩量与涨跌的配合度、OBV趋势 |
| 波动位置 volatility_position | 15% | 布林带%b位置、乖离率(是否短期透支/超跌) |
| 相对强弱 relative_strength | 15% | 相对基准指数(默认沪深300)的20/60日超额收益 |

趋势权重最高,因为这是"中长期"分析的核心——本skill不是给日内
博弈用的,趋势方向错了,其他维度再好看也意义有限。

如果基准指数数据获取失败,relative_strength维度会被跳过,
其权重按比例分配给其余4个维度(代码中已实现,无需手动调整)。

## 分档规则

每个维度的具体加减分规则见 `score.py` 源码注释,这里只说汇总后的分档:

| 加权总分 | 结论 |
|---|---|
| ≥ 1.0 | 技术面偏多,可考虑买入/逢低布局 |
| 0.3 ~ 1.0 | 技术面温和偏多,信号强度一般,建议分批/谨慎参与 |
| -0.3 ~ 0.3 | 技术面中性,信号不明确,建议观望 |
| -1.0 ~ -0.3 | 技术面温和偏空,建议规避或减仓 |
| ≤ -1.0 | 技术面明显偏空,建议规避/逢高减仓 |

## 置信度(高/中/低)的判定

置信度不是"分数绝对值有多大",而是"有多少个独立维度指向同一个方向"。
- **高**:|总分|≥0.8 且维度一致率≥80%(大多数维度同向)
- **中**:|总分|≥0.3 且维度一致率≥60%
- **低**:其余情况——包括分数虽然不小但维度严重分裂(比如趋势强烈
  向好但动量/量能强烈背离),这种情况即使总分不低也应该判为低置信度,
  并在报告里明确指出"指标之间存在矛盾"

## 风险否决规则(优先级高于打分结果)

以下情况会**强制覆盖**打分结论,改为"观望"并把置信度压到"低":
- 标的带有ST/*ST标记(财务/退市风险,技术分析对其参考价值显著下降)
- 历史数据不足60个交易日(次新股或长期停牌,均线/趋势计算不可靠)
- 数据中存在疑似长期停牌缺口(技术形态被打断)

以下情况**不会**覆盖结论方向,但会把置信度压到"低"并要求在报告中
显著提示:
- 最近20个交易日内出现3天以上接近涨跌停(波动极端,指标可能失真)

这条规则背后的原则是:**当数据本身不可靠时,模型的分析能力再强也无法
弥补,诚实地说"现在不是给出可靠技术结论的好时机"比硬给结论更负责任。**

## 止损/止盈参考的计算逻辑

止损给两个候选,不强制二选一,而是在报告里说明权衡:
- **支撑位止损**:最近的关键支撑位(摆动低点聚类/近20-250日新低中,
  价格下方最近的一档)——更贴合"价格结构",但如果支撑位离当前价
  很远,止损空间会偏大。
- **波动率止损(2倍ATR)**:收盘价-2×ATR14——更贴合"正常波动幅度",
  避免被随机噪音打掉,但在弱势股上可能止损位会显得"过宽"。

止盈/目标位同理给两个候选(最近阻力位 / 收盘价+3×ATR14)。

风险回报比(risk_reward_ratio_illustrative)只用其中一组(优先取
支撑位止损+阻力位止盈,缺失则用ATR版本)做示意性计算,
报告中应说明这只是参考,不是精确预测,实际操作中用户应结合自己的
风险承受能力选择止损/止盈位置,而非照搬某个数字。

## 如果用户想自定义权重

如果用户明确说"我更看重量能"/"我不关心和大盘比较"之类的偏好,
可以临时调整 `DEFAULT_WEIGHTS` 字典里的数值重新跑一遍 `score.py`
(权重之和应为1,或让代码里已有的归一化逻辑处理),并在报告里
注明"本次分析使用了用户自定义权重: ...",不要默默修改默认值
影响后续所有分析。
